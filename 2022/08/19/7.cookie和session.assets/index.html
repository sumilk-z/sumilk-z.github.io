Java使用Cookie、Session和Filter

Author: zhu Date: August 19, 2022  10:56:19 Category: Java

会话

会话:用户打开浏览器，访问web服务器的资源，会话建立，直到有一方断开连接，会话结束。在一次会话中可以包含==多次==请求和响应。

会话跟踪:一种维护浏览器状态的方法，服务器需要识别多次请求是否来自于同一浏览器，以便在同一次会话的多次请求间==共享数据==

为什么浏览器和服务器之间不支持数据共享：

  • 浏览器和服务器之间使用的是HTTP请求来进行数据传输
  • HTTP协议是==无状态==的，每次浏览器向服务器请求时，服务器都会将该请求视为==新的==请求
  • HTTP协议设计成无状态的目的是让每次请求之间相互独立，互不影响
  • 请求与请求之间独立后，就无法实现多次请求之间的数据共享

Cookie

客户端会话技术，将数据保存到客户端，以后每次请求都携带Cookie数据进行访问。

cookie是由服务端创建的

cookie的使用场景

假如在一次会话中有两个请求，第一个请求服务端处理的时候新建一个cookie对象然后写入一些数据。在响应客户端的时候将cookie也发送到客户端，客户端接收到之后就保存在本地（浏览器中）。浏览器第二次发起请求的时候会携带这个cookie对象的数据。

Cookie的原理分析

  • tomcat服务器基于HTTP协议来相应数据
  • 第一个servlet中，tomcat会在响应头中添加：Set-Cookie:username=jack
  • 浏览器获取到响应结果之后，从响应头中读取数据，并将数据存储在浏览器的内存中。
  • 当浏览器再次发起请求的时候，浏览器会自动在请求头中添加：Cookie:username=jack发送给服务端对应的servlet
  • 这样servlet就可以从request对象中获取到Cookie对象列表以及具体的数据了。

Cookie使用细节

Cookie存活时间

默认情况下：cookie存储在浏览器内存中，当浏览器关闭，内存释放，则cookie被销毁。

可以设置cookie存活时间

  • 正数：将cookie写入浏览器所在电脑的硬盘，持久化存储，到时间删除。
  • 负数：默认值，当浏览器关闭，cookie被销毁
  • 0：立刻删除对应的cookie

Session

session: 服务端会话跟踪技术，将数据保存到服务端。

  • Session存储在服务端，而Cookie存储在客户端
  • 存储在客户端的数据容易被窃取和截获，存在很多不安全的因素
  • 存储在服务端的数据相比于客户端来说就更安全

特别需要注意：session也是基于cookie来实现的。

session原理

Session要想实现一次会话多次请求之间的数据共享，就必须要保证多次请求获取Session的对象是同一个。

这种保证是由cookie来提供的。

如果是不同浏览器或者重新打开浏览器后，打印的Session就不一样了。

所以Session实现的也是一次会话中的多次请求之间的数据共享。

关键：session的id存储在cookie中

  • 首先访问ServletSetSession,会获取一次session，得到一个session对象，该session对象有唯一的ID
  • 在ServletSetSession的response中，把session对象的ID写入到cookie对象中
  • 浏览器接收到response之后，把cookie中的数据保存到浏览器的内存中。
  • 浏览器再次发起访问ServletSetSession的时候，把cookie对象的数据放到响应头中一起请求，这其中就包括这次会话所建立的session的唯一ID。
  • 服务器接收到cookie之后从中读取到本次会话的session的id,再次获取session的时候就能获取到之前相同对象的session.

session的钝化和活化

钝化：服务器正常关闭之后，tomcat会自动将session数据写入服务器的硬盘中。这就是钝化。

注意是正常关闭，非正常关闭不保证数据能正常保存下来。

活化：当再次启动服务器后，从文件中加载数据到session中。

数据加载到session中之后，原来存储session的文件会被删掉，以保证session数据是最新的。

那么，在服务器重启的过程中，只要客户端没有关闭浏览器，即还是在一次会话的过程中，那么依旧可以获取到正常的session数据。

应用场景：用户在添加购物车的过程中服务器重启，购物车中的数据不会丢失。

Filter

过滤器可以把客户端的访问进行过滤。Filter 表示过滤器，是 JavaWeb 三大组件(Servlet、Filter、Listener)之一

Filter的使用

  • 定义类，实现Filter接口，重写其中的方法
  • 配置Filter拦截资源的路径：在类上定义@webFilter注解，注解的value属性表示拦截的资源/*表示拦截所有的资源

拦截路径有如下四种配置方式：

  • 拦截具体的资源：/index.jsp：只有访问index.jsp时才会被拦截
  • 目录拦截：/user/*：访问/user下的所有资源，都会被拦截
  • 后缀名拦截：*.jsp：访问后缀名为jsp的资源，都会被拦截
  • 拦截所有：/*：访问所有资源，都会被拦截

filter就像是过滤网，只要是访问的URL符合其value值范围的，都要执行经过该filter，filter可以不止一个。

Ajax

AJAX (Asynchronous JavaScript And XML)：异步的 JavaScript 和 XML。

ajax的作用体现在

  • 与服务器进行数据交换：通过AJAX可以给服务器发送请求，服务器将数据直接响应回给浏览器
  • 异步交互：可以在==不重新加载整个页面==的情况下，与服务器交换数据并==更新部分网页==的技术，如：搜索联想、用户名是否可用校验，等等…

ajax使得实现前后端分离成为可能：

​ ajax是客户端和服务端之间的通信工具，服务端可以直接将数据发送到客户端。而之前使用的JSP是服务端将数据嵌入到HTML文档中，然后发送整个HTML文档到客户端。

Axios 对原生的AJAX进行封装，简化书写。

Axios官网是：https://www.axios-http.cn

  • 引入axios的文件

  • 发送get请求

axios() 是用来发送异步请求的，小括号中使用 js 对象传递请求相关的参数：

  • method 属性：用来设置请求方式的。取值为 get 或者 post。
  • url 属性：用来书写请求的资源路径。如果是 get 请求，需要将请求参数拼接到路径的后面，格式为： url?参数名=参数值&参数名2=参数值2。
  • data 属性：作为请求体被发送的数据。也就是说如果是 post 请求的话，数据需要作为 data 属性的值。

then() 需要传递一个匿名函数。我们将 then() 中传递的匿名函数称为 ==回调函数==，意思是该匿名函数在发送请求时不会被调用，而是在成功响应后调用的函数。而该回调函数中的 resp 参数是对响应的数据进行封装的对象，通过 resp.data 可以获取到响应的数据。

为了方便起见， Axios 已经为所有支持的请求方法提供了别名。
